# ToDining — Council Code Review

> Generated by an adversarial multi-agent audit — **218 agents**: 11 specialist finders → a **prosecutor vs. defender** argued every finding, with a **judge** breaking ties → synthesis. Date: 2026-07-14.
>
> **83 findings confirmed** (of 93 raised; 10 rejected by the council, 20 required a judge).

## Verdict: `BLOCKED` for production

ToDining is a feature-complete, well-architected SaaS prototype sitting on a fundamentally unsafe security foundation. The product surface is impressive — QR ordering, live kitchen/waiter boards, billing, inventory, reservations, analytics, and multi-tenant switching all work — and the data layer is a genuinely clean, swappable abstraction. But nothing that protects data actually enforces anything. Every access gate is client-side decoration: the /admin console renders the full management suite for unauthenticated visitors (RoleGuard `open` + DashboardLayout's anon=owner default), the app never uses Supabase Auth so the RLS policies keyed on auth.uid() are inert, the anon key can read and write every tenant's rows, and the admin password hash is world-readable to anyone holding the public anon key. Staff and admin sessions are both forgeable from localStorage with no credential re-validation, passwords are hashed with non-cryptographic djb2 under a shared salt, and hardcoded fallback credentials plus credential-hash console logging ship in the browser bundle. Beneath the security tier sits a second, quieter class of correctness problems: money/tax inputs saved unvalidated, ID generation that can collide and silently overwrite rows, order state machines that allow backward transitions on billed orders, destructive deletes with no confirmation, and a realtime model whose cross-device alerts and update indicators simply never fire in Supabase multi-device mode. The CLAUDE.md is refreshingly honest about all of this — the team knows. This is a strong demo and a poor production candidate; the gap is not features, it is that the entire trust boundary still lives in the client.

## Key themes (root causes)

- Client-trust security model with no server-side enforcement: every auth gate (RoleGuard, staff session, admin session) is client-side only, RLS is keyed on an auth.uid() that is never set, and secrets (admin hash, fallback creds) live in the public bundle — so all authentication and tenant isolation is effectively cosmetic.
- No real tenant isolation or data-minimization: the whole multi-tenant database is loaded into the browser, any session can switch to any tenant, public INSERT policies use `with check (true)`, and the Supabase realtime subscription is global and unscoped — one tenant's client sees and can mutate all tenants' data.
- Missing guardrails on destructive and state-changing actions: single-click deletes with no confirmation/undo (tables, menu, inventory, staff), unclamped tax/service-% inputs that corrupt every future bill, arbitrary/backward order-status transitions on billed orders, and no protection against removing the last owner.
- Realtime correctness only works single-device: alerts, ticket age, sidebar update dots, and optimistic writes all ride the in-app bus, but cross-device Supabase events don't feed those paths — so in the real multi-device deployment staff silently miss new-order/service/low-stock signals.
- Accessibility gaps are systemic because they live in shared primitives: Modal has no focus trap/labelling, contrast failures are baked into design tokens (ink-muted, ember-500 labels, red-500 errors), form errors aren't wired to aria attributes, and there's no reduced-motion support — so the whole admin surface inherits the same defects.
- Weak engineering hygiene / quality gate: the only 'lint' (tsc --noEmit) type-checks zero files and always passes, there are no tests or error boundaries, and production ships verbose console logging of full order/customer payloads and credential hashes.

## Top priorities

1. **Stand up real server-side auth and close the /admin open door**
   - *Why:* This is the single gating item for production. Remove the RoleGuard `open` bypass and DashboardLayout's anon=owner default, move staff and admin auth onto Supabase Auth (server-verified sessions, not a forgeable localStorage id), then enable RLS and re-apply policies. Until auth exists server-side, every other security fix is cosmetic because the anon key still has full read/write.
   - *Effort:* Large (multi-week; touches auth contexts, routing, Supabase provisioning, RLS rollout — must land auth before RLS or the app locks out entirely)
2. **Kill the credential-exposure criticals**
   - *Why:* Independently exploitable today with only the public anon key: revoke anon SELECT on admin_users (currently `using(true)` exposes the password hash), delete the hardcoded LOCAL_FALLBACK credentials from the bundle, remove the credential/hash console logging in adminAuthService, and replace djb2 hashing (handled for free once auth is server-side). Rotate the committed creds.
   - *Effort:* Small–Medium (mostly deletions + one SQL grant revoke + credential rotation; low code risk, high payoff)
3. **Scope data loading and realtime to the active tenant**
   - *Why:* Even with auth, loading the entire multi-tenant DB into the client and subscribing to all postgres_changes globally leaks and fans out cross-tenant data and causes full-table refetches on every device. Scoping queries and the realtime channel by restaurant_id is both a security control and the fix for the multi-device alert/update-dot failures.
   - *Effort:* Medium (store.ts subscription + query scoping; coordinate with the realtime bus paths)
4. **Harden money math and data integrity**
   - *Why:* These silently corrupt records with no attacker needed: replace makeId() with crypto.randomUUID to stop cross-device collisions and silent upsert overwrites, clamp/validate tax and service %, enforce a forward-only order state machine so billed orders can't be reopened, and fix the concurrent Complete-Dining invoice race that drops the persisted bill.
   - *Effort:* Medium (localized service-layer changes; the billing layer is already carefully structured, so mostly validation + a UUID swap + a transactional guard)
5. **Fix the shared UI primitives (broad blast radius, cheap)**
   - *Why:* A handful of primitive-level fixes correct defects across the whole app at once: Button size=lg uses the non-existent h-13 class so every large CTA collapses (one-line fix), Modal needs a focus trap/initial-focus/restoration and labelling (fixes accessibility across every admin form), and the contrast tokens (ink-muted, ember-500 labels, red-500 errors) plus form aria wiring fix WCAG failures everywhere they're used.
   - *Effort:* Low–Medium (concentrated in Button.tsx, Modal.tsx, Input.tsx, tailwind.config.js; disproportionate coverage per hour)
6. **Make the quality gate real**
   - *Why:* The project currently has no gate that can catch regressions: fix `npm run lint` so tsc actually type-checks the source, configure esbuild `drop` to strip console logging (which today ships PII payloads to prod), and add top-level error boundaries plus a route errorElement so a single throw doesn't blank the app. This protects every fix above from silently regressing.
   - *Effort:* Low–Medium (config + a small ErrorBoundary; enables everything else to be verified)

## What is genuinely well-built

- The data-layer abstraction is genuinely well-designed: UI imports only from typed services backed by a synchronous in-memory store with localStorage persistence, Supabase write-through, and an in-app realtime bus — swappable between mock and Supabase with no UI changes. This is the strongest part of the codebase and real production-grade thinking.
- The billing domain shows careful money discipline: session billing is idempotent, invoice numbers derive from MAX(sequence) rather than a row count (so deletes never renumber or reuse), totals are summed from per-order values to avoid rounding drift against Analytics, and a withItems() rehydration path reconstructs line items lost on Supabase round-trips. The author clearly reasoned about correctness, not just happy-path.
- Documentation honesty is exceptional: CLAUDE.md and the security-remediation doc enumerate the exact critical holes, in code, with an ordered runbook (auth-before-RLS to avoid lockout) — the team is not hiding risk, which materially lowers the cost of fixing it.
- Sound architectural hygiene at the structure level: feature-first layout with thin route components, route-level lazy() code-splitting so the customer QR path never ships recharts/jsPDF/admin code, a consistent cn()-composed design-system primitive set, zod+react-hook-form validation, and dense rationale comments throughout.
- TypeScript-strict intent and a coherent, tasteful design system (warm 'modern dining' palette, Fraunces/Jakarta pairing, defined tokens for radii/shadows/animation) give the product a polished, consistent feel that would be expensive to retrofit — the remaining UI work is corrective, not foundational.

## Confirmed findings

### 🔴 Critical (4)

#### C1. /admin renders the full management console for unauthenticated visitors because RoleGuard `open` plus DashboardLayout's anon=owner default grant owner access to anyone
**Area:** Security, Auth & Multi-tenancy · **Category:** auth-bypass · **Location:** `src/app/router.tsx:84` · **Confidence:** 0.97

- **Failure scenario:** An attacker (or any curious user) with no login and empty localStorage types /admin into the address bar. RoleGuard's `!user && open` branch (RoleGuard.tsx:28) returns the children without redirecting to /login. DashboardLayout defaults the missing user to role 'owner' (line 35), so the full owner sidebar — including owner-only Analytics and Restaurants — renders. Navigating to /admin/orders, /admin/billing, /admin/staff, /admin/restaurants loads each admin page and displays live data across tenants, with write actions available. The /admin-panel AdminPanelGuard login is entirely bypassed because it never wraps /admin.
- **Fix:** Remove the `open` prop from the /admin RoleGuard (router.tsx:84) so unauthenticated visitors hit RoleGuard.tsx:29 and are redirected to /login. Additionally remove the `user?.role ?? 'owner'` fallback in DashboardLayout.tsx:35 (and the anon "Staff sign in" fallback path in UserChip) so the console never renders defensively as owner without an authenticated staff session whose role is owner|manager. If direct-open behavior is genuinely wanted for the panel, route /admin through AdminPanelGuard instead of `open`, so a validated login is still required.

#### C2. admin_users grants anon SELECT with `using (true)`, exposing the admin password hash to anyone holding the public anon key
**Area:** Security, Auth & Multi-tenancy · **Category:** credential-exposure · **Location:** `supabase/migrations/0008_admin_panel_credentials.sql:34` · **Confidence:** 0.97

- **Failure scenario:** Attacker obtains the public anon/publishable key (shipped in the frontend bundle, by design) and runs `curl -H "apikey: $ANON" -H "Authorization: Bearer $ANON" "$SUPABASE_URL/rest/v1/admin_users?select=*"`. RLS `using(true)` returns the row: `{"username":"SAMP-IP(manoj)","password_hash":"19q822l"}`. The attacker brute-forces the unsalted-per-user djb2 hash (or simply guesses the short, human plaintext `mavoc-2026`) in milliseconds, then signs into /admin-panel as the platform operator. Alternatively, since adminAuthService compares the hash client-side, the attacker patches the in-browser check to bypass login entirely. Either path grants full Admin Panel access.
- **Fix:** Never send credentials to the browser. Revoke `select on public.admin_users` from anon (and drop the `using(true)` policy); keep the row readable only by service_role. Move Admin Panel authentication to a server-side Supabase Edge Function / RPC that runs with service_role, verifies the password server-side, and returns only a short-lived signed session token — the browser never sees the hash. Replace djb2 with a real, per-credential-salted algorithm (bcrypt/argon2) computed server-side, and rotate the currently-exposed `mavoc-2026` credential since it must be treated as compromised.

#### C3. RLS policies are keyed on auth.uid() but the app never uses Supabase Auth, and public INSERT policies use `with check (true)`, so the anon key can read/write every tenant's data
**Area:** Security, Auth & Multi-tenancy · **Category:** tenant-isolation · **Location:** `supabase/policies.sql:81` · **Confidence:** 0.95

- **Failure scenario:** An attacker extracts the anon key from the shipped JS bundle and calls the Supabase REST endpoint directly. `GET /rest/v1/bills?select=*` returns all 5 billing rows across all 6 tenants; `GET /rest/v1/staff?select=*` and `/rest/v1/customers` leak staff and customer PII cross-tenant (RLS currently off). Independently — and this survives even after RLS is turned on — `POST /rest/v1/orders` with a body specifying an arbitrary victim restaurant_id succeeds because orders_insert_public uses `with check (true)`, injecting forged orders/service-requests/reservations/feedback into any competitor's live boards. Full cross-tenant read (until RLS on) and cross-tenant write (persists past RLS).
- **Fix:** Follow docs/SECURITY-REMEDIATION.md in order: (1) provision Supabase Auth users for existing staff via service_role and populate staff.auth_uid; (2) wire login to supabase.auth.signInWithPassword and defer bootstrapStore() hydration until a session exists so the initial SELECT runs as the authenticated user; (3) only then `enable row level security` on all tables and re-apply policies.sql; (4) re-run the anon probe to confirm 0 rows. Separately and regardless of auth, replace `with check (true)` on the five public INSERT policies with constraints that (a) require restaurant_id to reference a real restaurants row, (b) for order_items require the parent order to exist, and ideally (c) validate against a resolved table/slug from the QR path plus server-side rate limiting, so anonymous customers can only insert into a legitimately-scoped tenant.

#### C4. Staff session is forgeable via a predictable localStorage staff id, yielding a full owner session without credentials
**Area:** Security, Auth & Multi-tenancy · **Category:** auth-bypass · **Location:** `src/context/AuthContext.tsx:21` · **Confidence:** 0.9

- **Failure scenario:** Attacker (or any curious user) opens devtools on any deployment, runs `localStorage.setItem('todining_auth_staff','stf_owner')` and reloads. AuthProvider rehydrates `user` to the seeded owner of restaurant rA with role 'owner' — no password prompted or checked. RoleGuard for /admin, /kitchen, /waiter passes, DashboardLayout renders the full management console (orders, billing, staff, inventory, analytics, restaurants), and every owner-gated action is available. Because ids like stf_owner/stf_mgr/stf_owner_b are guessable and getById returns the full record, this needs zero prior knowledge of any credential.
- **Fix:** Stop deriving the authenticated user from a client-writable id. Adopt Supabase Auth: on load, read the verified session (`supabase.auth.getSession()`), and resolve the Staff record from the JWT-bound `auth.uid` (staff.auth_uid) rather than from a localStorage id. Hydrate the store only after a real session exists, enable RLS so `auth.uid()` gates rows, and remove the guessable seeded ids from any production build. As a minimum interim hardening, persist a server-signed token (not a bare id) and re-validate it on load, but the id-in-localStorage pattern should be removed entirely. This is item 1-4 of the CLAUDE.md production runbook.

### 🟠 High (14)

#### H1. Password hashing is djb2 (non-cryptographic, 32-bit, shared static salt), making stored hashes trivially reversible
**Area:** Security, Auth & Multi-tenancy · **Category:** weak-crypto · **Location:** `src/lib/password.ts:17` · **Confidence:** 0.97

- **Failure scenario:** An attacker with the public Supabase anon key (shipped in the client bundle) issues `select username, password_hash from admin_users` — permitted by the anon SELECT policy from migration 0008 — and retrieves `SAMP-IP(manoj)` / `19q822l`. They recover the plaintext either by reading migration 0008's own comment (`mavoc-2026 → 19q822l`) or by enumerating the ~4.29-billion-entry 32-bit djb2 space (seconds on commodity hardware; the hash function is public in password.ts). They then sign in at /admin-panel as the platform operator. The same applies to owner accounts (cafe@2026, velans@2026), whose plaintext is likewise committed. No cracking is even strictly required since verification is client-side.
- **Fix:** Move all credential verification server-side onto Supabase Auth (bcrypt/argon2 with per-user salts, JWT sessions, RLS keyed to auth.uid()); the anon client must never see password_hash. Remove the anon SELECT policy on admin_users so hashes are not world-readable. Immediately rotate every credential ever committed (mavoc-2026, cafe@2026, velans@2026) since they are permanently in git history. Purge all plaintext passwords from migration comments and seed source, and delete the console.info that logs generated hashes. Retire src/lib/password.ts entirely rather than swapping djb2 for a stronger client-side hash — client-side hashing plus client-side comparison is not authentication regardless of algorithm.

#### H2. Button size="lg" uses the non-existent Tailwind class h-13, so every large primary CTA collapses to text-line height (shorter than a small button)
**Area:** UI / Visual Design & Design-System Consistency · **Category:** design-system · **Location:** `src/components/ui/Button.tsx:27` · **Confidence:** 0.97

- **Failure scenario:** A guest on a phone opens the cart. The fixed bottom-bar 'Place order' button (CartPage.tsx:77, size=\"lg\" fullWidth) renders at the text line-box height (~24px tall) instead of the intended ~52px, because h-13 emits no CSS and there is no vertical padding. The button is a squashed full-width strip below the 44px tap-target minimum, with the loading spinner/label vertically cramped. The same collapse hits the LoginPage and AdminPanelLogin 'Sign in' buttons and the LandingPage hero CTAs — every large primary CTA in the app.
- **Fix:** Change the lg size string to use real, self-contained classes. Simplest and consistent with the rest of the scale: `lg: 'h-12 px-6 text-base gap-2'` (h-12 = 3rem = 48px is a valid default class), or use padding-based height `lg: 'px-6 py-3 text-base gap-2'`. Alternatively add `spacing: { 13: '3.25rem' }` to theme.extend in tailwind.config.js if 52px is specifically desired. Whichever is chosen, run a build (or check the generated CSS) to confirm the height class actually emits. Note md/sm are fine (h-11/h-9 are valid), so only the lg entry needs fixing.

#### H3. Password-less Manager quick-card grants full admin-console access with no credential
**Area:** Security, Auth & Multi-tenancy · **Category:** auth-bypass · **Location:** `src/pages/LoginPage.tsx:120` · **Confidence:** 0.95

- **Failure scenario:** A user opens the deployed app at /login. Without typing anything, they tap the 'Manager' quick-card (which already shows the manager's name and email). signIn(member.email) runs with password undefined; staffService.authenticate returns the passwordHash-less manager record; AuthContext sets the user and persists their id to localStorage. navigate(ROLE_CONFIG['manager'].home) lands them on /admin/orders. RoleGuard passes them because canAccess(manager, '/admin/...') is true. They now view and modify orders, tables & QR codes, menu, categories, reservations, inventory, billing and staff for the active tenant — with no credential ever entered. The session survives reload via the AUTH_KEY in localStorage.
- **Fix:** Require a real credential for any role that can reach /admin. Concretely: (1) give the seeded manager rows a passwordHash and drop the password-less manager quick-card, or gate the Manager card behind the same password form owners use; (2) harden staffService.authenticate so it does not sign in privileged roles (any role whose ROLE_CONFIG.allows includes '/admin') without a verified password — the 'return staff when !passwordHash' fallback should be restricted to non-privileged demo roles (waiter/kitchen) at most, and ideally removed entirely before production. Longer term, authorization must move server-side (real Supabase auth + RLS) rather than relying on client-side RoleGuard/ROLE_CONFIG checks against a mock store.

#### H4. Cross-device staff alerts (new order / service call / order ready / low stock) never fire in Supabase multi-device mode
**Area:** State, Store & Realtime · **Category:** realtime-correctness · **Location:** `src/data/mock/store.ts:140` · **Confidence:** 0.95

- **Failure scenario:** Supabase configured; kitchen tablet and guest phone are separate browser sessions. Guest at Table 5 places an order on their phone. Phone tab emits order:created locally (irrelevant to kitchen). Kitchen tablet receives the new row via Postgres realtime -> subscribeRealtime schedule('orders') -> refetch('orders') -> emits only data:changed('orders'). The kitchen board's useLiveQuery (subscribed to data:changed) re-renders and the order appears in the list, but the useRealtime alert subscription (types:['order:created']) never receives an order:created event, so the '🔔 New order — Table 5' toast (5s duration) never plays. Identical failure for the waiter's 'service:created' call-button alert and the 'order ready' toast, and for inventory:low warnings — every cross-device staff alert is silently lost while the lists still update.
- **Fix:** Translate row-level Supabase changes into semantic events in the realtime ingestion path rather than collapsing everything to data:changed. In refetch(), diff the previous cache against the freshly fetched rows for the affected table: newly-present orders -> emit order:created with {orderId, tableNumber}; orders whose status changed -> emit order:updated with {orderId, status}; new service_requests -> service:created; inventory rows crossing lowThreshold -> inventory:low. Emit these with the row's real restaurantId (keeping the data:changed broadcast for list refresh). Capture the before-snapshot before overwriting db[spec.dbKey] at store.ts:138. Alternatively (less clean) make the alert consumers diff previous vs current lists on data:changed to detect newly-arrived/newly-ready orders, but the emit-semantic-events approach keeps consumer code unchanged and fixes all four alert types at once.

#### H5. Modal has no focus trap, no initial focus move, and no focus restoration, so keyboard/screen-reader users lose their place and can tab into the inert background
**Area:** Accessibility (WCAG) · **Category:** a11y · **Location:** `src/components/ui/Modal.tsx:31` · **Confidence:** 0.95

- **Failure scenario:** A keyboard-only or screen-reader user on the customer flow activates 'Help', which opens ServiceRequestSheet via this Modal. Because nothing moves focus into the dialog, focus remains on the Help button behind the overlay; the dialog is announced (if at all) as an unnamed "dialog" with no title. Pressing Tab does not cycle the four request buttons — it walks into the underlying menu/cart controls that are still in the tab order and not inert. The user cannot reliably reach or confirm 'Request Water'. On dismiss, focus is not restored, so they are dumped at the top of the page. The same failure repeats in every admin CRUD modal (StaffForm, MenuItemForm, InventoryForm, etc.), where a save/delete form is keyboard-unreachable without leaking into the page behind it.
- **Fix:** In Modal.tsx: (1) capture the previously focused element and, on open, move focus to the dialog container (add a ref + tabIndex={-1} and call ref.focus() in an effect, or focus the first focusable child); restore focus to the saved element on cleanup/close. (2) Implement a Tab/Shift+Tab wrap trap scoped to the dialog by querying focusable descendants and cycling at the boundaries. (3) Generate ids for the title and description and wire aria-labelledby/aria-describedby on the role="dialog" element (conditionally, only when title/description exist). (4) Mark the app root inert (or aria-hidden="true") while the modal is open so background content is removed from the tab order and the accessibility tree. Because this is a single shared primitive, one fix here remediates all 41+ call sites at once; consider adopting a vetted library (e.g., focus-trap-react or Radix Dialog) to avoid edge cases.

#### H6. Admin Panel session is forgeable: AdminAuthContext restores a session from localStorage checking only an expiry timestamp, never re-validating credentials
**Area:** Security, Auth & Multi-tenancy · **Category:** auth-bypass · **Location:** `src/context/AdminAuthContext.tsx:29` · **Confidence:** 0.9

- **Failure scenario:** Open the app, then in DevTools console run: `localStorage.setItem('todining_admin_panel_session', JSON.stringify({username:'attacker', expiresAt: Date.now()+1e12}))` and navigate to /admin-panel. On mount, AdminAuthProvider's useState initializer (line 50) calls readSession(), which sees expiresAt in the future and returns the forged object; authenticated becomes true; AdminPanelGuard renders the full workspace manager. No credential is ever entered and adminAuthService.authenticate is never invoked.
- **Fix:** Stop treating a client-writable localStorage object as proof of authentication. Move the trust boundary to the server: issue a real session on successful login (Supabase Auth JWT / server-signed token), have the guard validate it against the backend on load, and enforce admin authorization with Supabase RLS/row policies on the admin_users and workspace tables so that forging a client flag grants no data access. At minimum in the current architecture, the localStorage blob must carry a server-issued signed token that cannot be reconstructed client-side — but the durable fix is server-enforced authorization, since the anon-key client model means UI gating alone can never be trusted.

#### H7. Hardcoded admin credential fallback is shipped in the browser bundle
**Area:** Security, Auth & Multi-tenancy · **Category:** credential-exposure · **Location:** `src/data/services/adminAuthService.ts:22` · **Confidence:** 0.9

- **Failure scenario:** A build deployed without VITE_SUPABASE_* (the app's supported localStorage/offline fallback mode) reaches line 72. Any visitor who reads the public source/bundle — or simply notices the credential in the repo — navigates to /admin-panel and signs in as SAMP-IP(manoj) / mavoc-2026, gaining the workspace/admin manager role. Additionally, because the seeded Supabase admin_users row mirrors this same credential, the same username/password may also authenticate against a Supabase-backed production deployment unless the DB hash was independently rotated.
- **Fix:** Remove the LOCAL_FALLBACK object and the plaintext comment entirely. Perform admin authentication server-side (Supabase Auth or an edge function with bcrypt/argon2), never a client-evaluated equality check. If an offline dev shortcut is truly needed, gate it behind an explicit import.meta.env.DEV flag (or a dedicated dev-only env var) so it is tree-shaken out of production builds, and require the developer to supply the credential via env rather than shipping a literal. Separately, rotate the Supabase admin_users row seeded by migration 0008 since its plaintext is now disclosed in the repo, and replace the djb2 demo hash with a real password hash.

#### H8. Inventory auto-deduction and low-stock alerts are completely inert on the Supabase backend because menu-item recipes are never persisted or hydrated
**Area:** Data Integrity & Money Math · **Category:** data-integrity · **Location:** `src/data/supabase/mappers.ts:83` · **Confidence:** 0.9

- **Failure scenario:** Restaurant runs on the Supabase backend. The project was seeded on a prior boot, so on every subsequent session bootstrapStore() calls hydrateAll(), which SELECTs menu_items and maps them through fromRow — returning each MenuItem with recipe=undefined. A customer places 500 Chilli Chicken orders (seed recipe: 0.2 kg chicken each, inv_a_chicken starting at 12 kg). In orderService.placeFromCart, menuService.getItem('mi_wings').recipe is undefined, so the `menuItem?.recipe?.length` guard is false, deductForRecipe is never called, inventory_items.stock_qty stays at 12 kg, and no inventory:low realtime event is emitted. The manager dashboard shows 12 kg chicken in stock while the kitchen has physically run out — with zero low-stock warning ever raised.
- **Fix:** Make recipes round-trip through Supabase. Preferred: add a menu_item_ingredients child-table spec mirroring the order_items pattern — a menuItemIngredientsToRows(item) that emits {id, menu_item_id, inventory_item_id, qty} rows, a fromRow reattach step (like attachOrderItems) that repopulates each MenuItem.recipe during hydrateAll and refetch, and include the child table in FK order after menu_items and inventory_items. Simpler alternative if a JSON column is acceptable: add `recipe: m.recipe ?? []` to menuItems.toRow and `recipe: row.recipe ?? undefined` to fromRow, backed by a jsonb `recipe` column on menu_items. Either way, add coverage so recipe survives a pushAll→hydrateAll cycle, since there is no test suite to catch this regression.

#### H9. Removing a table with active orders proceeds with no confirmation and orphans those orders
**Area:** Admin Dashboard · **Category:** data-integrity · **Location:** `src/pages/admin/TablesPage.tsx:129` · **Confidence:** 0.9

- **Failure scenario:** Table 5 is occupied with a pending order (status !== 'completed'). Admin taps the table tile, opening the modal which displays "1 active order(s)" and the order id. Admin clicks "Remove table" once. tableService.remove runs synchronously: db.tables loses Table 5 and db.qrCodes loses its QR. The modal closes with a green "Table 5 removed" toast. The pending order still sits in db.orders with tableId pointing at the deleted table; the Kitchen/Orders boards still show it (no order event was emitted to prompt reconciliation), but there is no table to seat/serve and the guest's QR scan link now 404s / cannot be re-scanned to add items. No undo is offered.
- **Fix:** Two-part: (1) In the UI, gate the Remove button behind a confirmation step that names the table and its active-order count (reuse the Modal component or a window.confirm at minimum), and consider disabling/warning when activeOrders.length > 0. (2) In tableService.remove, refuse or explicitly handle removal when the table has non-completed orders — e.g. check orderService.byTable(restaurantId, id).some(o => o.status !== 'completed') and either block with a thrown/returned error surfaced as a toast, or cascade by cancelling those orders and emitting an order event so all boards reconcile. #### H10. The entire multi-tenant database is loaded into the client and any authenticated session can switch to or query any tenant

#### H14. makeId() collides across devices/reloads, and Supabase upsert then overwrites the colliding row (silent data loss)
**Area:** Data Integrity & Money Math · **Category:** correctness · **Location:** `src/lib/id.ts:7` · **Confidence:** 0.8

- **Failure scenario:** During a dinner rush, two guests at different tables each open the QR menu on their phones (each is a fresh page load, so each client's counter resets to 1000). Both order objects also carry text primary keys. The write-through queue upserts them into the `orders` table (setup.sql: text PK); the second upsert overwrites the first row entirely — its items, totals, and session_id — with no error raised. Table A's order silently becomes a duplicate of Table B's; Table A's bill and its contribution to analytics disappear. The same id can also collide in order_items and sessions.
- **Fix:** Replace the counter/timestamp scheme with a collision-resistant generator. Simplest: `export function makeId(prefix='id'){ return `${prefix}_${crypto.randomUUID()}`; }` — a UUID string stores cleanly in the existing `text` PK columns and needs no schema change. If a shorter/readable id is desired, append real entropy with a delimiter, e.g. combine Date.now() with `crypto.getRandomValues(new Uint32Array(2))` rendered in base36 and separate segments with `_` so the stamp/entropy boundary is unambiguous. Never rely on a module-level counter that resets on reload for cross-client uniqueness. Note the finding's "match the schema's uuid PKs" rationale is off — the live schema is setup.sql with text PKs — but randomUUID() is still the correct fix and is forward-compatible if the uuid-based schema.sql is ever adopted.

**Area:** State, Store & Realtime · **Category:** perf · **Location:** `src/data/mock/store.ts:65` · **Confidence:** 0.85 · **⚖️ Judge-decided**

All of that is true.

1) Only the structuredClone at line 65 is synchronous on the main thread. 3) The cost is architecturally subsumed. This same code already loads every table across all tenants into browser memory via SELECT * and hands the whole Database to the client (anon key). Net: a real, correctly-described inefficiency in the swappable mock store, but by-design, deferred off the interactive path, and trivial at intended scale. It is a legitimate low-severity efficiency note worth a cheap improvement, not a medium defect.
